# kontynuuj

Rozbudujmy ten system o **konkretny LLM‑agent** i **dodatkowy tool** w MCP‑Skills, żeby cały flow był kompletny:

- dodamy `compute_metrics_for_repo` (metryki całego repo),
- oraz prosty LLM‑agent w Pythonie z `mcp-client`, który:
    - pobiera listę plików z MCP‑Git,
    - wywołuje MCP‑Skills,
    - zwraca sugestię refactoringu.

***

## 1. Nowy tool: `compute_metrics_for_repo`

Zmodyfikuj `server.py` MCP‑Skills, dodając nowy tool:

```python
COMPUTE_METRICS = Tool(
    name="compute_metrics_for_repo",
    description="Compute high‑level metrics for a given repo.",
    inputSchema={
        "type": "object",
        "properties": {
            "repo_id": {"type": "string", "description": "Logical repo ID, e.g. 'my_org/repo'"},
            "base_path": {
                "type": "string",
                "description": "Base path where repos are cloned, e.g. /repos",
                "default": "/repos",
            },
        },
        "required": ["repo_id"],
    },
)


async def compute_metrics_for_repo(self, call: Call) -> CallResult:
    repo_id = call.request.params["repo_id"]
    base_path = call.request.params.get("base_path", "/repos")

    repo_ = Path(base_path) / repo_id

    if not repo_.exists():
        return CallResult(
            callId=call.request.id,
            result=json.dumps({"error": f"Repo '{repo_id}' does not exist at {repo_}"}),
        )

    total_files = 0
    total_lines = 0
    total_imports = 0

    for file_path in repo_.rglob("*.py"):
        total_files += 1
        try:
            with file_path.open("r", encoding="utf-8", errors="ignore") as f:
                content = f.read()
            lines = content.splitlines()
            total_lines += len(lines)
            total_imports += sum(
                1 for line in lines if line.strip().startswith("import") or line.strip().startswith("from")
            )
        except Exception:
            pass

    metrics = {
        "repo_id": repo_id,
        "file_count": total_files,
        "total_lines": total_lines,
        "total_imports": total_imports,
    }

    return CallResult(
        callId=call.request.id,
        result=json.dumps(metrics),
    )
```

Zarejestruj go w `main()`:

```python
server.add_tool(
    COMPUTE_METRICS.name,
    sk.compute_metrics_for_repo,
)
```

Teraz LLM może wywołać:

```json
{
  "tool": "compute_metrics_for_repo",
  "params": {
    "repo_id": "my_org/repo"
  }
}
```

i dostać:

```json
{
  "repo_id": "my_org/repo",
  "file_count": 73,
  "total_lines": 8423,
  "total_imports": 412
}
```

### 2.2 `agent.py` – agent z MCP‑Git + MCP‑Skills

```python
import asyncio
from typing import Dict, Any
from mcp.client import MCPClient
from mcp.types import Call, CallResult

# 1. Konfiguracja MCP‑serwerów

MCP_SERVERS = {
    "git": "http://mcp-git-server:8080/mcp",
    "skills": "http://mcp-skills-server:8080/mcp",
}


async def main():
    # Uruchom klientów MCP
    git_client = MCPClient(MCP_SERVERS["git"])
    skills_client = MCPClient(MCP_SERVERS["skills"])

    await git_client.init()
    await skills_client.init()

    repo_id = "my_org/repo"
    branch = "main"
    module_path = "src/module_x"

    # 1. Pobierz listę plików z MCP‑Git
    list_files_req = Call(
        id="call-1",
        method="list_files",
        params={
            "repo": repo_id,
            "branch": branch,
            "path": module_path,
        },
    )
    list_files_resp: CallResult = await git_client.request(list_files_req)

    file_list = list_files_resp.result["files"]  # Zakładam, że API list_files zwraca {"files": [...]}
    print("Found files:", file_list)

    # 2. Wywołaj MCP‑Skills: analiza konkretnego modułu
    analyze_call = Call(
        id="call-2",
        method="analyze_code_structure",
        params={
            "repo_id": repo_id,
            "paths": file_list,
            "base_path": "/repos",
        },
    )
    analyze_resp: CallResult = await skills_client.request(analyze_call)

    analysis = analyze_resp.result
    print("Analysis:", analysis)

    # 3. (symulacja LLM) – tu możesz wstawić prompt do realnego modelu
    #    np. OpenAI / Claude / lokalny model, działa tak:
    #    - serializuj `analysis` do JSON,
    #    - zbuduj prompt: "Na podstawie tych danych zaproponuj refactoring...",
    #    - Wywołaj model.
    #
    # Przykład pseudopromptu:
    prompt = f"""
Na podstawie analizy repo:
{analysis}

Zaproponuj konkretny refactoring dla modułu {module_path}:
- które pliki warto rozbić,
- które zależności importu są nieoptymalne,
- jaki pattern architektoniczny zastosować.
Odpowiedz w postaci krótkiego planu (3–5 punktów).
"""

    print("\n=== LLM‑prompt ===")
    print(prompt)

    # 4. (opcjonalnie) – jeśli chcesz, żeby LLM zwrócił gotowe patche / edycje,
    #    możesz potem wrócić do MCP‑Git:
    #    - wywołać `read_file` dla konkretnego pliku,
    #    - zastosować patch z LLM,
    #    - `git_add`, `git_commit`, `git_push`.

    await git_client.close()
    await skills_client.close()


if __name__ == "__main__":
    asyncio.run(main())
```

## 3. Jak połączyć to wszystko w Docker‑compose

Wróć do `docker-compose.yml` i dodaj service dla LLM‑agenta:

```yaml
services:
  # ... (mcp-git-server, mcp-skills-server)

  llm-agent:
    build: ./llm-agent
    environment:
      OLLAMA_HOST: http://ollama:11434  # lub inne URL‑e LLM
    depends_on:
      - mcp-git-server
      - mcp-skills-server
    command: ["python", "agent.py"]
```

Teraz cały flow:

- `mcp-git-server` i `mcp-skills-server` działają w Docker‑ze,
- `llm-agent`:
    - łączy się z MCP‑Git,
    - łączy się z MCP‑Skills,
    - podejmuje pierwszy krok decyzyjny (prompt do LLM) na podstawie danych z obu serwerów.

## 4. Co możesz zrobić dalej

- **Dodać realny model**:
    - zastąp `print(prompt)` wywołaniem np. `openai.ChatCompletion.create`, Claude SDK, czy lokalnego modelu (Ollama, Llama.cpp).[^1][^2]
- **Dodać cykl „analyze → propose → commit”**:
    - jeśli LLM zwróci plan, agent może:
        - wywołać `read_file` z MCP‑Git,
        - zastosować zmiany,
        - umieścić modyfikowane pliki jako `git add` → `git commit` → `git push`.[^3][^4]
- **Zrobić „MCP‑Gateway wrapper”**:
    - np. w Go/Rust, który:
        - rutowań tool‑calls do różnych MCP‑serwerów,
        - loguje wszystkie interakcje,
        - ogranicza dostęp do repo (np. tylko wybrane branche).

***

Jeśli chcesz, w kolejnym kroku mogę:

- napisać konkretny handler dla **przetwarzania planu refactoringu** (np. LLM zwraca JSON z listą edycji, a agent je aplikuje w repo za pomocą MCP‑Git),
- albo przygotować plik `deploy.sh`, który:
    - buduje obrazy,
    - uruchamia `docker compose up`,
    - oraz testuje flow (np. `curl` po MCP‑skills i MCP‑git).
