# tak

Super, zrobimy to „konkretnie”: przygotuję Ci:

- minimalistyczny `docker-compose.yml` z:
    - MCP‑Git‑server (np. github‑MCP),
    - własnym MCP‑Skills‑serverem (Python),
    - i prostym MCP‑LLM‑agentem,
- oraz **przykładowy MCP‑Skills server** z jednym tool‑em `analyze_code_structure`, który potrafi:
    - otrzymać `repo_id` i listę ścieżek,
    - wczytać pliki z Git‑a (lub z lokalnej kopii),
    - policzyć podstawowe metryki (ilość plików, LOC, zależności importów),
    - zwrócić JSON do LLM‑a.

***

## 1. Przykładowy `docker-compose.yml`

```yaml
version: '3.8'

services:
  mcp-git-server:
    image: ghcr.io/github/github-mcp-server
    environment:
      GITHUB_PERSONAL_ACCESS_TOKEN: ${GITHUB_PAT}
      GITHUB_ORGANIZATION: ${GITHUB_ORG}
    ports:
      - "8081:8080"
    restart: unless-stopped

  mcp-skills-server:
    build: ./mcp-skills
    ports:
      - "8082:8080"
    volumes:
      - ./repos:/repos  # lokalne klony repo
    restart: unless-stopped

  mcp-llm-agent:
    image: openai/assistant-mcp-example:latest  # lub własny obraz
    environment:
      MCP_SERVERS: |
        {
          "git": "http://mcp-git-server:8080/mcp",
          "skills": "http://mcp-skills-server:8080/mcp"
        }
    ports:
      - "8083:8080"
    restart: unless-stopped
```

Używasz go tak:

```bash
export GITHUB_PAT="..."
export GITHUB_ORG="my-org"
docker compose up -d
```

- Git‑MCP: port 8081 → `http://host:8081/mcp`
- Skills: port 8082 → `http://host:8082/mcp`
- LLM‑agent: port 8083 → konfigurujesz go jako MCP‑client.

### 2.3 `server.py` – MCP‑Skills z `analyze_code_structure`

Poniższy przykład używa uproszczonego MCP‑servera (np. `mcp-server` SDK):

```python
from mcp.server import Server, Transport
from mcp.types import (
    Call,
    CallResult,
    Notification,
    InitializationResult,
    Resource,
    Tool,
)
from pathlib import Path
import json
from typing import Dict, Any, List

# 1. Tools spec

ANALYZE_CODE_STRUCT = Tool(
    name="analyze_code_structure",
    description="Analyze code structure of a given repo and paths.",
    inputSchema={
        "type": "object",
        "properties": {
            "repo_id": {"type": "string", "description": "Logical repo ID (e.g. 'my_org/repo')"},
            "paths": {
                "type": "array",
                "items": {"type": "string"},
                "description": "List of relative paths within repo",
            },
            "base_path": {"type": "string", "description": "Base path where repos are cloned, e.g. /repos", "default": "/repos"},
        },
        "required": ["repo_id", "paths"],
    },
)


class MCPSkillsServer:
    def __init__(self):
        self.repo_base = Path("/repos")

    async def analyze_code_structure(self, call: Call) -> CallResult:
        repo_id = call.request.params["repo_id"]
        paths = call.request.params["paths"]
        base_path = call.request.params.get("base_path", "/repos")

        repo_path = self.repo_base / repo_id
        repo_ = Path(base_path) / repo_id

        results = []

        for rel_path in paths:
            full_path = repo_ / rel_path
            if not full_path.exists():
                results.append({
                    "path": rel_path,
                    "exists": False,
                    "error": "File not found in repo",
                })
                continue

            with full_path.open("r", encoding="utf-8", errors="ignore") as f:
                try:
                    content = f.read()
                except Exception as e:
                    content = ""

            lines = content.splitlines()
            line_count = len(lines)
            import_count = sum(1 for line in lines if line.strip().startswith("import") or line.strip().startswith("from"))

            results.append({
                "path": rel_path,
                "exists": True,
                "line_count": line_count,
                "import_count": import_count,
                "first_few_lines_preview": lines[:5],
            })

        return CallResult(
            callId=call.request.id,
            result=json.dumps({
                "repo_id": repo_id,
                "path_analysis": results,
            }),
        )


# 2. Server glue

async def main():
    server = Server(
        name="mcp-skills",
        version="0.1.0",
        tools=[ANALYZE_CODE_STRUCT],
    )

    sk = MCPSkillsServer()

    server.add_tool(
        ANALYZE_CODE_STRUCT.name,
        sk.analyze_code_structure,
    )

    transport = Transport(mode="stdio")  # HTTP / Streamable HTTP lepiej dla produkcji
    await server.run(transport)


if __name__ == "__main__":
    import asyncio

    asyncio.run(main())
```

Ten serwer:

- nakłada się na repozytorium sklonowane w `/repos/my_org/repo` (możesz potem zintegrować z MCP‑Git, który pobiera repo),
- zwraca proste metryki (ile linii, ile importów) dla podanych ścieżek.

## 3. Flow dla LLM (przykład konkretny)

Założenia:

- repo: `my_org/repo`, branch `main`
- LLM ma dostęp do MCP‑Git i MCP‑Skills

Flow:

1. MCP‑Git – `list_files`:
```json
{
  "tool": "list_files",
  "params": {
    "repo": "my_org/repo",
    "branch": "main",
    "path": "src/module_x/"
  }
}
```

2. LLM dostaje listę plików:
    - `src/module_x/a.py`
    - `src/module_x/b.py`
3. LLM wywołuje MCP‑Skills:
```json
{
  "tool": "analyze_code_structure",
  "params": {
    "repo_id": "my_org/repo",
    "paths": [
      "src/module_x/a.py",
      "src/module_x/b.py"
    ],
    "base_path": "/repos"
  }
}
```

4. MCP‑Skills zwraca:
```json
{
  "repo_id": "my_org/repo",
  "path_analysis": [
    {
      "path": "src/module_x/a.py",
      "exists": true,
      "line_count": 230,
      "import_count": 12,
      "first_few_lines_preview": ["import os", "import logging", "..."]
    },
    {
      "path": "src/module_x/b.py",
      "exists": true,
      "line_count": 180,
      "import_count": 8,
      "first_few_lines_preview": ["from a import X", "..."]
    }
  ]
}
```

5. LLM:
    - łączy dane,
    - podejmuje decyzję: „rozbij moduł X na dwa mniejsze, przenieś część logiki do osobnego pliku”.

## 4. Co możesz rozbudować dalej

- MCP‑Skills:
    - dodaj `compute_metrics_for_repo` (cyclomatic complexity, test coverage, cyclomatic complexity per file),
    - `recommend_patterns_from_repo` (np. patterny, które często się powtarzają w projekcie).
- MCP‑Git ↔ MCP‑Skills:
    - automatycznie klonuj repo w MCP‑Skills przy pierwszym `repo_id`, jeśli jeszcze nie istnieje.
- MCP‑Gateway / Bash:
    - `deploy.sh`:
        - `docker compose down`
        - `git pull` (z repo z ustawieniami)
        - `docker compose up -d`
        - `curl $LLM_AGENT_URL/health` + prosty smoke test.

***

Jeśli chcesz, w kolejnym kroku mogę:

- napisać konkretny `tool`‑schema dla `compute_metrics_for_repo` w MCP‑Skills (z przykładem metryk),
- albo przygotować prosty LLM‑agent (np. w Pythonie) z `mcp-client`, który samodzielnie wywołuje cykl `list_files → analyze_code_structure → propose_refactoring`.
